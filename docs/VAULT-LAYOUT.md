# Vault layout (the excluded data layer)

> **Sanitized note.** This document describes the structure of the private data
> vault. The vault itself is **not** included in this repository — it holds real
> third-party contact data, live pipeline state, and operational records. Only
> the directory names and their purpose are documented here, so a reader can
> understand how the code that *is* included expects its data to be organized.
> No real data appears in this file.

The system of record is a plain-file "vault" (a directory tree of Markdown and
JSON, browsable as an Obsidian vault). It is the single source of truth: the
Python engine and the console read and write files here; there is no database.
Each area keeps its own generated `INDEX.md`, and one canonical ownership map
assigns exactly one writer-agent to each area.

## Top-level areas

| Directory | Purpose | Writer (role) |
|---|---|---|
| `state/` | Rendered board and engine outputs (`board.json`, parity/audit reports, dirty-flags). The console reads these. | truth engine / Valence |
| `pipeline/` | Operational send state: cadence ledger + schedule, gated draft packs, sent records, send-approvals, audit run logs. | Mailman |
| `inbox/` | Reply state: per-lead threads, classification verdicts, orphan replies. | Archivist |
| `companies/` | Engaged-company pages (registrable-domain unit), regenerated by the truth engine. | Archivist |
| `clients/` | Human-written dossiers for engaged leads (interaction logs, context). | operator / Archivist |
| `leads/` | Lead universe: `raw/`, `verified/`, `rejected/`, `off-icp/`, `deferred/`, dedupe audits, generated leadbook. | Scraper |
| `meetings/` | Meeting notes and scheduling state. | Archivist |
| `rfps/` | Bid/RFP program: source maps, `raw → qualified → applying → submitted → won/lost` lifecycle dirs, portal registration packets. | Bids |
| `reference/` | Living specs and policy: ICP definition, product catalog, gate rules, draft-example golden voice, runbooks, DNC list. | Steward / operator |
| `suppression/` | Do-not-contact state (email-level, JSONL). | operator / gates |
| `audits/` | Dated, append-only audit reports. | Steward |
| `metrics/` | Rollup counters and periodic metrics. | tools |
| `os/` | The OS layer: agent ownership manifest, the LLM shell (`os/llm-shell/`, included in this repo), schedule logs, and the backup mirror. | Valence |

## Key conventions

- **One writer per state file.** Every file has exactly one tool allowed to
  write it; everything else reads. A second writer is treated as a bug. The
  full map is in [`STATE-CONTRACT.md`](STATE-CONTRACT.md).
- **Truth is external where it must be.** What was actually sent lives in the
  mail provider's Sent box, not in a file; the vault caches derive from it and
  are always regenerable.
- **Registrable-domain is the company unit.** Two mailboxes at the same
  registrable domain are one company; two strangers at a freemail domain are
  not (freemail keys by full address).
- **Records are append-only.** Sent records and audit logs are backfilled from
  external truth, never invented.
- **Secrets are never vaulted.** Credentials live outside the vault and are
  excluded from every backup and mirror (see `ops/velab_vault_push.sh`).

## What the included code expects

- `core/` (the truth engine) reads the mail corpus and writes `state/board.json`
  plus parity/audit reports under `state/`.
- `console/` (the operator UI) reads `state/` and `pipeline/` and shells out to
  the engine and workspace tools; its data roots are configurable via
  environment variables (`VELAB_VAULT`, `VELAB_WORKSPACE`, ...).
- `llm-shell/` skills and commands operate over `leads/`, `pipeline/`, and
  `rfps/` under the same ownership rules.

Because the data layer is excluded, the code in this repository is presented for
study, not as a turnkey deployment. Standing the system up would require
providing a vault of this shape plus mail-provider and verification credentials.
